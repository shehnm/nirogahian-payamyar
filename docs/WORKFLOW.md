# قانون ثبت تجربه — اجباری

> هر کسی روی این پروژه کار می‌کند (انسان یا اجنت) **باید** قبل/بعد از کار طبق این سند عمل کند.  
> هدف: تکرار بن‌بست نشود، مسیر به تحویل کارفرما از دست نرود.

---

## چه چیزهایی حتماً ثبت می‌شود

| رویداد | کجا ثبت شود | اجباری |
|--------|-------------|--------|
| پچ / تغییر کد mu-plugin | `CHANGELOG.md` + `PROJECT-MEMORY.md` | بله |
| تست روی سرور (موفق یا ناموفق) | `CHANGELOG.md` | بله |
| تست **ناموفق** / سایت down / rollback | `ROADMAP.md` جدول Dxx | بله |
| تست **موفق** (ایده کار کرد) | `ROADMAP.md` بخش «تجربه‌های موفق» Txx | بله |
| ترفند / نکته deploy (مثلاً فقط Upload نه paste) | `ROADMAP.md` یا این فایل | بله |
| مکالمه جدید کارفرما | `CLIENT-REQUEST.md` + `CHANGELOG.md` | بله |
| تغییر رمز / FTP | `ACCESS.md` (لوکال) | بله |
| ایده‌ای که **نزدیم** چون شبیه بن‌بست بود | `ROADMAP.md` — یادداشت کوتاه در D یا زیر «الگوهای ممنوع» | توصیه‌شده |

**قانون طلایی:** اگر روی سرور یا در کد ریپو چیزی عوض شد و در docs نیامد = کار نیمه‌کاره است.

---

## قبل از هر کار

1. بخوان: [ROADMAP.md](ROADMAP.md) — بن‌بست‌ها (D01–…)
2. بخوان: [PROJECT-MEMORY.md](PROJECT-MEMORY.md) — وضعیت فعلی
3. بخوان: [CLIENT-REQUEST.md](CLIENT-REQUEST.md) — هدف کارفرما
4. اگر پچ: چک کن ایده شبیه Dxx نیست

---

## بعد از هر کار (چک‌لیست commit)

```
[ ] CHANGELOG.md — تاریخ + یک خط (همه تست‌ها، حتی تست کوچک)
[ ] PROJECT-MEMORY.md — «وضعیت فعلی سرور» به‌روز
[ ] اگر شکست: ROADMAP.md — ردیف D جدید
[ ] اگر موفقیت مهم: ROADMAP.md — ردیف T جدید
[ ] اگر فایل پچ: mu-plugins/production یا development
[ ] git commit + push (بدون ACCESS.md)
```

قانون در `.cursor/rules/project-memory.mdc` با `alwaysApply: true` فعال است — هر سشن این workflow را رعایت می‌کند.

دستور به اجنت:

```
طبق docs/WORKFLOW.md ثبت کن — قبل از پچ ROADMAP را چک کن.
```
